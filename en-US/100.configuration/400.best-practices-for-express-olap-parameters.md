# Recommended configurations for OLAP

OceanBase summarizes the recommended configurations of core parameters and variables for various business scenarios based on its tuning experience from a large number of real scenarios. OceanBase Database can provide high performance in various business scenarios based on the recommended configurations. This topic describes the recommended configurations for online analytical processing (OLAP).

## Background information

OLAP is a data processing technology for multi-dimensional analysis of large-scale datasets. It allows you to perform complex analysis for data from different perspectives, including dimensions, measurements, limitations, and sorting. OLAP can help you identify and understand data patterns, trends, and correlations so as to support decision-making and business intelligence. OLAP is generally applied in data warehouses and business intelligence systems.

## Obtain a configuration template

OceanBase Database incorporates the templates of recommended configurations for different business scenarios in the installation package since V4.3.2. After you install the RPM package of OceanBase Database, you can find the configuration templates in the `/home/admin/oceanbase/etc` directory.

* `default_parameter.json` stores the recommended configurations of parameters.
* `default_system_variable.json` stores the recommended configurations of variables.

## Applicable version

The OLAP scenario applies only to OceanBase Database V4.3.2.

This section takes OceanBase Database V4.3.2 as an example to describe the recommended configurations for V4.3.x. You can view the recommended configurations for a version later than V4.3.2 in the configuration template for the corresponding version.

## Recommended parameter configurations

The following table describes the recommended parameter configurations for OLTP.

| **Parameter** | **Description** | **Default value** |  **Value range**  | **Recommended value** | **Level** | **Effective upon OBServer node restart** |
|------------|----------|------------|------------|--------------------------|----------|----------|
| enable_record_trace_log | Specifies whether to record trace logs. | True | <ul><li>True</li><li>False</li></ul> | False </br>This value specifies not to record trace logs, so as to improve analytical processing (AP) performance. | Cluster-level parameter | No |
| trace_log_slow_query_watermark | The threshold of the query execution duration. A query whose execution duration exceeds this threshold is considered a slow query. Trace logs of slow queries are written to system logs. | 1s. The default unit is microseconds. | [1ms, +∞) | 7d</br>This value specifies not to define slow queries in AP scenarios. | Cluster-level parameter | No |
| large_query_threshold | The execution time threshold to identify a large query. A request may be suspended if its execution time exceeds this threshold. A suspended request is considered a large query and is processed by following the large query scheduling strategy. | 5s | <ul><li>For Oceanbase Database V4.x earlier than V4.3.0: [1ms, +∞)</li><li>For Oceanbase Database V4.3.0 and later: [0ms, +∞)</li></ul> | 0ms</br>This value specifies not to detect large queries. | Cluster-level parameter | No |
| default_table_store_format | The default format for a table created in a user tenant, which can be a rowstore table, columnstore table, or hybrid rowstore-columnstore table. | row | ("row", "column", "compound")<main id="notice" type='explain'><h4>Note</h4><ul><li>`row`: specifies to create a rowstore table.  </li><li>`column`: specifies to create a pure columnstore table. If the <code>with column group</code> clause is not appended to the table creation statement, the <code>with column group(each column)</code> clause is automatically appended to the statement. </li><li>`compound`: specifies to create a hybrid rowstore-columnstore table. If the <code>with column group</code> clause is not appended to the table creation statement, the <code>with column group(all columns, each column)</code> clause is automatically appended to the statement. </li></ul></main> | column</br>This value specifies to create a pure columnstore table by default. | Tenant-level parameter | No |
| _rowsets_max_rows | The maximum number of rows processed in a vectorized execution by the SQL engine. | 256 | [1, 65536] | 256</br>This value is sufficient for conventional OLAP workloads. | Tenant-level parameter | No |

## Configuration examples

* Configure cluster-level parameters in the `sys` tenant.

    ```shell
    obclient> ALTER SYSTEM SET enable_record_trace_log = False;
    obclient> ALTER SYSTEM SET trace_log_slow_query_watermark = '7d';
    obclient> ALTER SYSTEM SET large_query_threshold = '0ms';
    ```

* Configure tenant-level parameters in a user tenant.

    :::tab
    tab MySQL tenant

    ```shell
    obclient> ALTER SYSTEM SET default_table_store_format = 'column';
    obclient> ALTER SYSTEM SET _rowsets_max_rows = 256;
    ```

    tab Oracle tenant

    ```shell
    obclient> ALTER SYSTEM SET default_table_store_format = 'column';
    obclient> ALTER SYSTEM SET "_rowsets_max_rows" = 256;
    ```

    :::

## Recommended variable configurations

You also need to specify variables while specifying parameters. The following table lists the recommended variable configurations.

| **Variable** | **Description** | **Default value** | **Recommended value** | **Effective scope** |
|----------|----------|------------|--------------------------|----------|
| ob_query_timeout | The maximum execution duration allowed for an SQL query. | 10000000, in microseconds | 604800000000, which indicates 7 days | <ul><li>Global</li><li>Session</li></ul> |
|  ob_trx_timeout | The transaction timeout period. | 86400000000, in microseconds | 604800000000, which indicates 7 days | <ul><li>Global</li><li>Session</li></ul> |
| parallel_degree_policy | The strategy based on which the optimizer selects the degree of parallelism (DOP). | MANUAL | AUTO, which specifies to enable the Auto DOP strategy. In other words, the optimizer automatically determine the DOP during plan generation. | <ul><li>Global</li><li>Session</li></ul> |
| parallel_min_scan_time_threshold | The minimum estimated execution duration of the base table scan. This variable is used in the Auto DOP strategy to calculate the DOP. When the estimated execution duration of the base table scan exceeds the specified value, parallel scan is enabled, and an appropriate DOP value is calculated based on the specified value. | 1000, in milliseconds | 10 <main id="notice" type='explain'><h4>Note</h4><p>When this variable is set to `10`, OceanBase Database can achieve optimal parallel execution performance for queries with execution duration of more than 100 ms. </p></main>| <ul><li>Global</li><li>Session</li></ul> |
| ob_sql_work_area_percentage | The maximum percentage of tenant memory for SQL execution. | 5 | 30 <main id="notice" type='explain'><h4>Note</h4><p>This variable is set to a large value to reduce the costs of memory leaks. </p></main>| Global |
| collation_server | The default character set and collation for the server. | utf8mb4_general_ci | utf8mb4_bin <main id="notice" type='explain'><h4>Note</h4><p>This collation can improve the performance by 20% compared with other collations. </p></main>| <ul><li>Global</li><li>Session</li></ul> |
| collation_connection | The character set and collation for the connection. | utf8mb4_general_ci | utf8mb4_bin <main id="notice" type='explain'><h4>Note</h4><p>This collation can improve the performance by 20% compared with other collations. </p></main>| <ul><li>Global</li><li>Session</li></ul> |

## Configuration examples

:::tab
tab MySQL tenant

```shell
obclient> SET GLOBAL ob_query_timeout = 604800000000;
obclient> SET GLOBAL ob_trx_timeout = 604800000000;
obclient> SET GLOBAL parallel_degree_policy = AUTO;
obclient> SET GLOBAL parallel_min_scan_time_threshold = 10;
obclient> SET GLOBAL ob_sql_work_area_percentage = 30;
obclient> SET GLOBAL collation_server = utf8mb4_bin;
obclient> SET GLOBAL collation_connection = utf8mb4_bin;
```

```shell
obclient> SET GLOBAL ob_query_timeout = 604800000000;
obclient> SET GLOBAL ob_trx_timeout = 604800000000;
obclient> SET GLOBAL parallel_degree_policy = AUTO;
```
